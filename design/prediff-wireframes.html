1 — Comment Lifecycle

The 5-state machine that governs every comment from creation to resolution. This is the load-bearing state model for prediff's feedback loop.

[ Draft ] ── "Send Feedback" ──▸ [ Submitted ] ── agent revises region ──▸ [ Addressed ] ── reviewer confirms ──▸ [ Resolved ] │ │ │ │ │ anchor deleted/unmatchable │ reviewer reopens │ ▼ │ │ [ Stale / Orphaned ] ◂───────────────────────┘ │ │ │ ├── re-anchor manually ──▸ [ Submitted ] │ ├── convert to file note ──▸ [ Submitted ] │ └── dismiss ──▸ [ Resolved ] │ └── autosaved locally (debounced ~1s, survives crash/reload)

Visual treatment per state

Draft Draft Dashed outline, muted. Only visible to reviewer. Autosaved locally.
Submitted Submitted Solid outline, no badge needed (this is the "normal" state). Persisted server-side.
Addressed Addressed Agent-accent badge. Agent changed the region — reviewer should check if it's right.
Needs attention Stale/Orphaned Caution-amber badge. Anchor is gone or can't be confidently relocated. Never silent.
Resolved Resolved Collapsed to one line with checkmark. Expandable. Thread visually shrinks.

Severity tags (optional per comment)

Single-select, not required. Signals priority to the agent without writing it out in the comment body.

2 — Review Summary / Send Feedback Flow

"Send Feedback" is the primary loop-closing action. It batches all Draft comments, submits them, and signals the agent to begin a new revision. This is the peak-end moment — per the visual direction, it gets more design care than its frequency warrants.

3 — Waiting for Agent / Agent is Revising

After "Send Feedback," the agent is working. The reviewer may still be reading other files. The key principle: never block the reviewer's current activity. The agent's working state is shown in the sync indicator and optionally in a non-blocking banner, not as a modal or loading spinner.

Top bar sync indicator states

Synced Steady state. All comments persisted, agent idle.
Saving... Brief, during comment autosave or submission. Neutral, not alarming.
Agent is revising Pulsing agent-accent dot. Feedback was received, agent is generating next revision.
Offline — reconnecting Connection lost. All local drafts preserved. Auto-retry in progress.
Sync failed — retry Rare. Submission failed. Affected comments revert to Draft with retry affordance.

"Agent is revising" — in-diff banner (non-blocking)

4 — Disconnected / Sync Failure States

The local server (launched by the agent) can die, crash, or lose connection. The design response: never leave feedback in an ambiguous state. Failed submissions visibly fail and revert to Draft with retry. Full disconnect gets an unmissable overlay, not a toast.

5 — Stale / Orphaned Comment Triage

When a new revision deletes or heavily rewrites the code a comment was anchored to, the comment becomes orphaned. It is never silently dropped and never silently reattached wrong. The reviewer explicitly resolves it.

6 — Session History / Revision List

Revisions are numbered and never deleted. The reviewer can view what changed between any two revisions as a diff-of-diffs. This is collapsed/optional — most reviews never need it, but it must be there for trust.

7 — Upstream Git Conflict

prediff is a review tool, not a merge tool. When the local branch diverges from origin in a way that creates a real git conflict, prediff surfaces it clearly and defers to the developer's editor/terminal. No auto-resolution.

Per-file conflict banner (blocking for that file only)
⚠
Upstream conflict detected
src/auth.py has upstream changes that may conflict with this diff.
Resolve in your editor or terminal before continuing review here.

8 — Agent Outside Stated Scope

If the agent was launched with a stated task (e.g. "fix the login bug") and touches files outside what's inferable from that scope, the file tree shows a small informational flag. Non-blocking — agents legitimately touch adjacent files sometimes — but it turns implicit anxiety into an explicit, glanceable answer.
