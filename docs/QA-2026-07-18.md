# Internal QA pass — 2026-07-18

First-pass QA executed by the LLM (per project experiment protocol), following
the scenario script prepared for the external QA task. Environment: macOS,
nix-built package (`nix build` output), Chromium-based browser pane, 1440×900.
Two test repos: a realistic 3-file "agent change" (pagination + caching, incl.
one untracked file) and a 32-file / ~13.5k-line migration diff (incl. one
3,200-line file and a lockfile).

## What was exercised, end to end

Cold open from the nix package (0.47s CLI→URL) · gutter-click and keyboard
(`c`) commenting, range composer, tag chips, draft autosave · drafts-private
semantics ("Send Feedback" batch → concurrent `prediff wait` woke with exit 2
and the exact batch) · agent-side `refresh`/`resolve --reply` appearing live
over SSE · revision banner (no view yank), revision spine, switching Rev 2 ↔ 4
· re-anchoring across 4 revisions (silent follow + correct `submitted`
retention when nearby code survives) · orphaning on region deletion → "1
NEEDS ATTENTION" badge → triage panel → convert-to-file-note · full state
survival across browser reload · disconnect overlay on daemon kill · Mark
Ready with unresolved-comments warning → `wait` exit 0 · untracked files in
the working diff · large-diff behavior: auto-collapse (deleted/generated/
lockfile/>800-line), "mark all viewed" bulk action, minimap, instant deep
scroll (scrollTop 200k/277k, ~90 DOM rows) · light + dark themes.

## Findings

### F1 — MAJOR: daemon restart strands open browser tabs
The daemon binds a random port per start. After `prediff stop` + `open`, the
session resumes (same session-id, disk state intact) but on a NEW port; the
already-open tab shows the Disconnected overlay promising "Reconnecting
automatically…" against a port that will never come back. The overlay's trust
promise is false in exactly the scenario it exists for.
**Repro**: open review in browser → `prediff stop` → `prediff open` → old tab
spins forever; reviewer must find the new URL in the terminal and loses their
place.
**Suggested fix**: persist the chosen port in the repo state dir and rebind it
on restart (fall back + update lockfile only if taken). Alternatively a tiny
well-known discovery endpoint; port reuse is simpler.

### F2 — MAJOR (trust): out-of-scope heuristic is wrong often enough to harm
Token-overlap scope matching produced: `db.py` flagged out-of-scope on a
caching task where db.py was the core file (repo 1), and 30 of 32 files
flagged on "migrate api endpoints to v2" (repo 2). An amber warning that is
usually wrong trains reviewers to ignore it — worse than absent, per the
design's own trust principle (spec §9.4 wanted this "glanceable answer", and
currently it answers wrong).
**Suggested fix**: require stronger evidence before flagging (path/directory
overlap, not just filename tokens), cap the flagged fraction (if >half the
diff is "out of scope" the signal is meaningless — suppress), and/or let the
agent pass expected paths (`--scope-files`). Demote visual weight until
precision is decent.

### F3 — MINOR: first "Mark Ready" click after page load is swallowed
Fresh page load → first click on Mark Ready does nothing; second click opens
the confirmation. Likely hydration/focus timing.

### F4 — MINOR: mouse↔keyboard focus handoff
After clicking a UI button (e.g. "Add draft"), keyboard hunk/file navigation
state is stale: `n` (next file) followed by `c` anchored a comment to the
previous file's context line. A reviewer mixing mouse and keyboard will
mis-anchor comments. Keyboard focus should re-sync to the last-interacted
location, and `n`/`p` should always move file focus regardless of prior
mouse activity.

### Observations (not defects)
- Flat file list (matches visual comp) instead of spec §2's directory tree —
  acceptable at moderate file counts, revisit for monorepo-scale reviews.
- Comments can anchor to blank context lines (valid, renders fine, slightly
  odd — API callers should prefer content lines).
- Browser-pane rAF freezing when unfocused blocked in-harness FPS tracing;
  scroll perf covered by the Playwright bench instead (0 long tasks @10k).

## Verdict

The core loop is genuinely usable and the novel parts (revision banner,
drafts-private batching, orphan triage, disconnect messaging) behave exactly
per the design spec — with F1 as the one real trust-breaker to fix before
calling the "feedback is never lost" promise honest, and F2 as the one place
the UI actively misleads. F3/F4 are polish. Everything else exceeded
expectations, including performance.

Fix order: F1 → F2 → F4 → F3.
